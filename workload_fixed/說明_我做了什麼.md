# 我做了什麼(白話版)

> 這份是給「回來看不知道發生什麼事」的你。一句話總結:
> **我把 headline 的測試資料,從「我們自己寫程式生的」換成「真的 YCSB 生的」,並產出了第一個正式檔 `YC`,而且證明實驗程式能跑它。**

---

## 1. 為什麼要做這件事?

之前所有 workload(丟給實驗跑的操作清單,像 `read 155`)都是我們**自己寫 `gen_workload.py` 生出來的**。

自己生有兩個風險:
- **容易不小心生出「剛好對自己有利」的資料**——之前 C 那個 −75% 的漂亮數字,後來發現是資料範圍設超過而製造出來的假熱點(不是真本事)。
- **審稿人不信**。你自己定的規則,別人沒辦法對照。

**解法**:headline(拿去講故事、上摘要的那組)改用學術界公認的標準工具 **YCSB** 來生。這樣資料的分佈是公認的,不是我們手調的。

`workload_fixed/README.md` 就是這個「怎麼換」的完整規格書(spec)。我這次是去**執行**它。

---

## 2. 遇到的第一個坎:這台電腦沒有 Java

YCSB 是 Java 寫的工具,但這台機器**沒裝 Java,也沒有管理員權限**(不能 `sudo apt install`)。

如果因為沒 Java 就「用 Python 假裝一個 YCSB」——那就**又變回自己生**,整件事白做。

所以我先解決這個坎:**在不需要權限的情況下**,把一份可攜式 Java + YCSB 的程式包下載、解壓到你的家目錄 `~/ycsb-tools/`(不是專案資料夾裡,不會弄髒 repo)。現在「**真的** YCSB」在這台機器上能跑了。

---

## 3. 我具體做了四件事

| # | 做了什麼 | 白話 |
|---|---|---|
| 1 | 裝好真 YCSB | 讓這台沒權限的機器能跑真 YCSB(見上面第 2 點) |
| 2 | **驗證 spec 的假設** | 那份 README 對 YCSB 內部行為講了很多「它應該會這樣」。我**實際跑 YCSB 一條條去對**,確認是真的還是猜的 |
| 3 | 做好一條生產線 | 一鍵把 YCSB 的輸出,轉成你系統吃得下的 workload 檔,中間自動檢查 |
| 4 | 產出第一個正式檔 `YC` | 並證明實驗程式真的讀得動、跑得起來 |

### 第 2 件事的結果:spec 大致是對的 ✅

我最擔心的一條(spec 說某個參數在某情況下「沒作用」)——**實測證實它真的沒作用**,所以 spec 後面的計畫不用大改。其他幾條假設也都對。

但有 **5 個「照著 README 做會出錯」的操作細節**我修好了(直接改進 README,旁邊標了 🔧)。例如:README 教你用的某個指令其實在這台跑不動、YCSB 的輸出裡混了三種不同的東西(不處理的話轉檔程式會誤判)。這些都是實務坑,不是 spec 的觀念錯。

> 想看完整驗證數據 → [`verification/VERIFICATION.md`](verification/VERIFICATION.md)

## 5. 產出的 `YC` 長怎樣?(體檢結果)

- 8 萬筆操作,**全部是讀取**。
- **命中率 100%**(不會像舊 C 那樣有一半在查不存在的資料 → 沒有假熱點)。
- 有**真實的熱點**(有些資料被反覆讀),但熱度**溫和**,不是我們手調出來的極端值。
- 熱點**穩定不移動**(這點有量化數字佐證,不是嘴巴講)。
- 拿去給實驗程式跑:**8 萬筆全部讀到、沒出錯**。

一句話:**這是一個乾淨、可信、來源公認的 headline workload**,把之前 C 的毛病都避開了。

---

## 6. ⚠️ 一個要提醒你的發現

我順手量了一下:`YC` 對**現在這個資料庫**來說,「要預取的骨架」只有 **204 KB**,非常小。

白話:**這麼小的東西,幾乎一定整個被電腦自動放進記憶體了**。那我們研究的「預取」還有什麼好預取的?**可能量出來效果趨近於零**。

這不是壞消息,也不是 bug——spec 本來就預想到這種情況。但意思是:**量產一整批之前,可能要先把資料庫弄大一點**(讓骨架大到不會整個進記憶體),否則跑完會發現 headline 沒效果。
