## 1. 背景與問題

- SQLite 把整個資料庫存成一個 **4 KB page 的陣列**，用 B+tree 組織。
- 每筆查詢都要**從樹根走到樹葉**，沿路的 interior page（內部節點）**全部都要在記憶體裡**。
- **冷啟動**（剛開機、cache 是空的）時，這些 page 都得從硬碟讀，每讀一個就是一次慢速的隨機 I/O。

**核心問題：能不能在第一筆查詢之前，先把這些關鍵 page 載進記憶體？**

---

## 2. 實驗設定

### 測試資料庫（固定一個，所有實驗共用）

| 項目 | 數值 |
|---|---|
| Page 大小 | 4 KB |
| 總筆數 | 600,000 rows |
| 總 page 數 | 26,331 |
| 整個 DB | ~102 MB |
| **Interior page（瓶頸）** | **92 個 → 368 KB（占 0.35%）** |
| Leaf page（資料本體） | 26,239 個 → ~102 MB（占 99.65%） |

**重點：interior 只占 0.35%（368 KB），但每筆查詢都得用到。只要先載這 368 KB，就能避開冷啟動的隨機 I/O。**

### 四種查詢情境（workload）

| 名稱 | 特性 | 像什麼 |
|---|---|---|
| **A** | 集中查少數熱門資料（Zipfian） | App 首頁、常開的聯絡人 |
| **B** | 平均亂查（uniform） | 隨機抽樣、爬蟲 |
| **C** | 只查最新加入的資料（檔尾） | 剛收到的訊息、剛拍的照片 |
| **D** | 寫入壓力產生器 | 模擬 DB 被持續修改 |

### 怎麼量

冷啟動 → 清空記憶體 cache → 執行 prefetch → 量第一筆查詢花多久。

## 3. 嘗試的做法

分三類，可以互相搭配：

| 類別 | 策略 | 做法簡述 |
|---|---|---|
| **改 layout** | 1a 原始 / 1b VACUUM / **1c 整理過** | 改變 page 在檔案裡的物理排列 |
| **Prefetch** | 2a–2c（看結構）/ 2d–2e（看歷史）/ 2f（抄 cache） | 第一筆查詢前先載哪些 page |
| **記憶體共用** | 多程式共用同一份 cache | 一個程式 prefetch，全部受惠 |

## 4. 主要結果

### 4.1 各情境最佳方法一覽

同一套量測基準（7 種方法 × 3 種 layout，A/B/C 同條件）下，每個情境表現最好的方法：

| 情境 | 特性 | 最佳方法 | 第一筆查詢 | 改善 |
|---|---|---|---:|---:|
| **A** | 熱門集中 | 抄上次 cache（記住上次熱了哪些 page，下次全載） | 305 → **16 µs** | **−95%** |
| **B** | 平均亂查 | 抄上次 cache | 464 → **17 µs** | **−96%** |
| **C** | 查檔尾新資料 | 抄上次 cache | 671 → **17 µs** | **−97%** |
| **D** | 持續寫入後（churn） | 看歷史 + 最熱 10 個 leaf node | 281 → **21 µs** | **−92%** |

> **「抄上次 cache」在沒有記憶體壓力時各情境通殺**，但它要記住最多 page（約 500 個）。
> 想省記憶體時，更輕量的方法也很划算：**A 用「prefetch 前 5 個 interior」(−54%)、
> C 用「看歷史只載用過的」(−48 ~ −83%)**，下面分項說明。

### 4.2 最佳組合（Workload A）

| 做法 | 第一筆查詢 | 改善 |
|---|---:|---:|
| 什麼都不做（baseline） | 318 µs | — |
| 只 prefetch 前 5 個 interior | 224 µs | **−30%** |
| **整理 layout + prefetch 前 5 個** | **127 µs** | **−69%** ← 結構式方法的最佳 |

### 4.3 不同情境差很多

| 情境 | 最好能改善多少 | 為什麼 |
|---|---:|---|
| **A**（熱門集中） | **−69 ~ −91%** | 葉子自然在 cache，只剩 interior 要救 |
| **B**（平均亂查） | −49% | 每筆都打到新葉子，救不掉 |
| **C**（查檔尾新資料） | −54 ~ −83% | 同上，但用「看歷史」的方法可突破 |

### 4.4 「看歷史」的方法最聰明（Workload C）

不是盲目載前 N 個，而是**先觀察哪些 page 真的被用到**，再只載那些：

| 做法 | 改善 | 載入次數 |
|---|---:|---:|
| 載全部 92 個 interior | −54% | 92 次 |
| **只載真正用過的 interior** | **−48%** | **4 次** ← 一樣效果，省 23 倍 |
| 再加最熱的 10 個葉子 | **−83%** | 14 次 |

---

## 5. 關鍵發現

1. **少即是多**：載前 5 個 interior（−54%）比載全部 92 個（−31%）還好——載太多反而來不及。
2. **沒有萬用解**：最適合載幾個 page，跟「資料怎麼排」「查詢什麼樣」強烈相關。
3. **整理 layout 對 A 是大勝（−69%），但對 B 反而變慢**——不能無腦套用。
4. **看歷史 > 看結構**：只載真正用過的 page，4 次載入就追平盲載 92 次的效果。
5. **動態環境下依然有效**：DB 被持續寫入（5 萬筆修改）後，效益完全沒衰退（A 仍 −91%、C 仍 −54%）。
6. **記憶體吃緊也撐得住**：限制到 20 MB，第一筆查詢的改善幾乎不受影響。
7. **多程式共用免費加倍**：一個程式做 prefetch，所有共用同一份 cache 的程式都受惠。

---

## 6. 實務建議

| 情境 | 建議做法 | 預期改善 |
|---|---|---|
| 熱門資料集中（最常見） | prefetch 前 5 個 interior | −54% |
| 想追求極致 | 先整理 layout，再 prefetch 前 5 個 | −69% |
| 平均亂查 / 查檔尾新資料 | 看歷史，只載用過的 + 最熱 10 個葉子 | −83% |
| 多程式共用 DB | 開共用記憶體，背景定時 prefetch | 成本固定、效益乘以程式數 |

## 總結

SQLite 開機後第一筆查詢很慢，因為要先從硬碟讀進那 **92 個關鍵的 interior page**。
我們用 **prefetch（提前載入）** 把它們先放進記憶體，最高可把第一筆查詢
**從 318 µs 降到 127 µs（−69%）**，而且這個方法在 DB 持續被寫入、記憶體吃緊、
多程式共用的情況下都站得住。
